# `->` 成员调用不回退到 efun/simul_efun 设计

## 背景

当前扩展在部分链路上会把 `obj->foo()` 中的 `foo` 当成普通函数名继续查找，最终回退到模拟函数库或系统函数库。这会造成 3 类错误行为：

1. 鼠标悬停在 `obj->foo` 上时，显示了与对象调用无关的 efun/simul_efun 文档。
2. 跳转定义时，在对象目标无法解析的情况下，错误跳到模拟函数库或系统函数库。
3. 在 `obj->` 后输入补全时，混入与对象成员无关的 efun/simul_efun 候选。

本次只修这个误回退 bug，不尝试建立完整的对象方法解析规则。

## 范围

本次变更只处理一个规则：

- 当调用形式是 `receiver->method(...)` 时，不允许把 `method` 继续当作 simul_efun 或 efun 解析。

本次不包含：

- 推断 `receiver` 的真实对象类型。
- 为 `new`、函数返回值、参数变量等对象来源建立完整解析链。
- 为 `this_object()->method()` 增加“回当前文件查找”的增强逻辑。
- 修改普通函数调用 `foo()` 的现有查找顺序。

## 目标行为

### Hover

- 如果当前光标位于 `receiver->method` 的 `method` 上，则禁止进入 simul_efun / efun 文档回退逻辑。
- 如果当前已有对象成员解析命中，则按对象成员结果返回。
- 如果对象成员无法解析，则返回空结果，不显示错误的 efun/simul_efun 文档。

### Definition

- `receiver->method()` 继续优先走现有对象访问定义跳转分支。
- 如果对象目标文件可解析，则只在该目标文件及该链路允许的对象相关位置中查找。
- 如果对象目标无法解析或方法不存在，则返回“找不到定义”。
- 不允许再回退到 simul_efun / efun 定义查找。

### Completion

- 当补全上下文为 `receiver->` 成员访问时，不混入 simul_efun / efun 补全项。
- 如果当前成员补全链路拿不到对象成员候选，则返回空或仅返回其他明确属于成员访问的候选。
- 普通标识符补全不受影响，仍可保留原有 simul_efun / efun 行为。

## 方案选型

### 方案 A：在三条链路分别加“成员访问禁止 efun 回退”判定

做法：

- hover 根据当前位置识别是否处于 `->` 成员名上。
- definition 复用已有对象访问检测结果；一旦命中成员访问分支，就禁止后续 efun 回退。
- completion 复用已有 `member` 上下文类型，在该上下文中排除 simul_efun / efun 候选。

优点：

- 改动最小。
- 与现有架构一致。
- 不需要提前解决复杂对象推断问题。

缺点：

- 三条链路各自有一层条件分支，规则会有少量重复。

### 方案 B：抽象统一的“调用类型判定器”

做法：

- 先统一判断当前位置属于普通函数调用还是成员调用。
- 再由 hover / definition / completion 共享该结果。

优点：

- 规则集中。

缺点：

- 对当前 bug 来说偏重。
- 需要重新梳理多个调用点，风险和改动面更大。

### 方案 C：开始做通用对象解析

做法：

- 为变量、返回值、内建对象函数等补建对象来源推断，再决定成员方法集。

优点：

- 长期能力更完整。

缺点：

- 明显超出本次 bugfix 范围。
- 容易把简单修复拖成大改。

## 采用方案

采用方案 A。

理由：

- 它精确对应当前已确认的问题边界。
- 可以先消除错误行为，再单独讨论通用 `对象->方法名` 解析规则。
- 不会把“禁止误回退”与“如何正确推断对象方法”两个问题耦合在一起。

## 组件影响

### `src/efun/EfunHoverProvider.ts`

- 在进入 simul_efun / efun 查找前，增加成员访问场景判定。
- 如果当前位置是 `receiver->method` 的成员名，则只允许返回对象相关结果；对象相关结果缺失时直接结束。

### `src/definitionProvider.ts`

- 保持现有对象访问分析入口不变。
- 明确对象访问命中后的失败语义：对象链解析失败即结束，不再进入 simul_efun / efun 回退。

### 补全链路

- 利用现有 `CompletionContextAnalyzer` 的 `member` 上下文。
- 在成员上下文下，屏蔽 simul_efun / efun 候选的注入。
- 不额外新增对象类型推断。

## 数据流

1. 用户把光标放在或输入到 `receiver->method`。
2. 当前链路先判断这是不是成员访问上下文。
3. 如果是成员访问：
   - 只走对象成员相关逻辑。
   - 不走 simul_efun / efun 兜底。
4. 如果不是成员访问：
   - 保持原有普通函数查找逻辑。

## 错误处理

- 无法识别对象目标时，返回空结果，而不是猜测成 efun。
- 目标文件存在但没有该方法时，返回空结果。
- 现有日志和缓存行为保持不变，避免引入无关副作用。

## 测试

需要补充或更新测试，覆盖以下场景：

1. `obj->foo()` 在 `foo` 同名 efun 存在时，hover 不显示 efun 文档。
2. `obj->foo()` 在无法解析对象目标时，definition 不跳到 simul_efun 或 efun。
3. `obj->` 补全时，不出现 simul_efun / efun 候选。
4. 普通 `foo()` 调用时，原有 simul_efun / efun 行为不变。

优先复用现有 provider / completion / efun 相关测试文件，避免新建无必要测试基础设施。

## 验收标准

- `receiver->method(...)` 不再显示 efun/simul_efun hover。
- `receiver->method(...)` 在对象解析失败时，不再跳到 efun/simul_efun 定义。
- `receiver->` 补全列表不再混入 efun/simul_efun。
- 普通函数调用链路行为保持不变。
