# 关于lpc开发范式的整理及lpc-support

目前在 LPC 开发这件事上，社区里其实已经有人在不同方向上做建设了。

- 有人在推更强的静态分析和语言服务
- 有人在推外部工具协同
- 有人在推更适合具体 Mudlib 场景的更新机制
- 也有人在推工作区真源、本地编译闭环、开发驱动、package 协议这一套东西

大家推进的切口不一样，落点也不一样，但大的方向其实是一致的：都在试着把 LPC 开发从过去那种高度依赖经验、依赖手工、依赖旧工程习惯的状态里往外推一点。

下面这篇文字，不是要替谁下定义，也不是要把哪条路线写成唯一正确答案。

它更像是对这件事的一次整理：一方面把当前这些方向放在一起看，另一方面把 `lpc-support` 这条路线背后的判断说清楚。

可以把它看成一些思想的参考表达，以及我自己的思考。

## 这件事先要说清楚一条边界

`lpc-support` 解决的不是生产环境问题。

它主要解决的是开发环境问题。

这句话看起来简单，但其实很重要。因为如果边界不先说清楚，后面很多判断都会跑偏。

这不是在做一个新的线上部署方案，也不是在定义一个唯一正确的 Mudlib。按我的理解，更重要的是先把开发这层基础重新立起来。

为什么是开发环境？

因为现在 LPC 真正长期欠账的，不是“线上跑不跑得动”，而是“开发这件事有没有一套像样的基础设施”。

过去很多东西都靠经验在撑：

- include 路径靠人记
- 模拟函数库路径靠人记
- 编译服务怎么接靠人记
- 哪个 driver 能拿来做开发靠人记
- 外部系统怎么装靠人记
- 出错以后去哪里找靠人记

这套东西短期能跑，但长期看往往会越来越重。项目越大，世界越大，内容越多，维护成本越高。

所以按我的判断，LPC 要继续往前走，不能只在内容层面做加法，开发环境这一层大概率还是要重建。

`lpc-support` 这条路线，基本就是沿着这个方向在展开。

## 可能不应该把“兼容旧 lib”放到最高目标

这件事我得说得直一点。

兼容旧 lib 很重要，我也一直在做兼容。但如果把“尽可能兼容一切旧结构”放成最高目标，新的技术基础大概率很难长出来。

原因不复杂。因为一旦最高目标变成兼容，所有设计都会往回收：

- 工程不能有新的真源，因为旧项目没有
- 工作区不能有明确配置，因为旧流程不是这么组织的
- driver 不能按开发环境需求重建，因为官方主线不是这么服务的
- package 不能有统一协议，因为旧工程没有这个概念
- 结构分析不能做清晰分层，因为旧代码不是按这个方式组织的

最后往往就会变成：工具一直在补，基础一直没变。

如果目标只是维护一套已经存在的世界，这样做还能撑很久。因为你只是在一套熟悉结构上继续往前堆内容。

但如果目标变了，问题就出来了。

按我的理解，真正值得解决的，不是“怎么继续在旧结构上堆一个世界”，而是“怎么让 LPC 有能力支撑更大的世界表达”。

这里说的“大”，不是地图更大，也不是设定文本更多，而是：

- 系统一致性更强
- 模块边界更清楚
- 外部能力更容易接入
- 工具反馈更可靠
- 协作摩擦更低
- 世界叙事不再只能靠局部拼接往前推

如果一直只以兼容旧 lib 为最高目标，这套底子大概率还是很难长出来。

这也是我为什么会说：`lpc-support` 想做的，不是一个旧工具增强版，而是一套新的 LPC 应用开发范式。

## 这套范式到底在重建什么

按我现在的判断，至少有三层东西值得优先重建。

### 1. 工程真源

过去很多 LPC 项目在工程层面其实没有真源。

配置是散的，路径是散的，工具链入口是散的，编译服务器是散的。编辑器知道一点，Mudlib 自己知道一点，开发者脑子里再记一点。

这在小项目里还能撑，在大项目里就会变成持续的隐性损耗。

所以 `lpc-support` 现在把工作区根目录的 `lpc-support.json` 作为第一入口。`config.hell` 通过它接进来，项目真正关心的结构信息通过它同步出来，编译模式、driver 配置、远程服务器、本地 `lpccp` 都通过它归拢到同一个地方。

这一步不是“多了一个配置文件”这么简单。

它真正的意义是：项目第一次开始有一个工程级真源。

一旦这个真源成立，后面很多东西才有可能统一：

- 编辑器理解项目
- 编译管理理解项目
- 错误树理解项目
- external package 理解项目
- AI 助手理解项目

如果没有这层真源，所有工具最后都会重新退回“各自猜各自的”。

### 2. 结构化分析

第二层，是编辑器对 LPC 的理解方式。

我不太想把 `lpc-support` 做成一个靠正则和缓存勉强维持的语言包。那种方式短期能出功能，长期看通常不太稳定。

所以现在主路径被拆成三层：

- `ParsedDocument`
- `SyntaxDocument`
- `SemanticSnapshot`

我刻意不用一个泛化的“AST”把它们糊在一起，就是因为职责本来就不一样。

- parser 层负责 token、trivia、基础诊断
- syntax 层负责稳定结构和 token-backed range
- semantic 层负责符号、继承、作用域和语义摘要

这不是为了概念好看，而是为了后面的能力能站住：

- 格式化不是靠文本猜
- 跳转不是靠局部扫描猜
- 补全不是靠缓存碰运气
- 错误定位不是靠“差不多在这”
- AI 工具拿到的是结构，不是碎文本

如果这层不稳，后面所有“现代化体验”都只是表面现象。

### 3. 本地开发闭环

第三层，是开发闭环。

传统 LPC 开发有个很大的问题：反馈链路太长。

你写完代码之后，很多时候要么去接远程接口，要么去看日志，要么去进环境里重编译，要么去猜到底是哪一步出了问题。这个过程不是完全不能用，但它不适合现代开发，也不适合大工程。

所以按这条路径往下走，更合理的目标应该是建立一个更短的本地闭环：

- 编辑器知道项目结构
- 本地 driver 就是开发驱动
- `lpccp` 直接连这个 driver
- 编译结果结构化返回
- 错误树能直接用
- 外部系统能力也能通过 package 安装进同一套工程

一旦这条链路打通，LPC 开发就不再只是“写代码，然后扔进旧环境里看结果”，而是开始有一点现代应用开发的样子。

## 这套东西现在是怎么组成的

如果从资源层面来讲，现在这套范式主要依赖三样东西：

- `lpc-support` 扩展
- 开发环境 driver 和 `lpccp`
- `external_system_package`

这三样东西不是并列摆件，它们是有分工的。

### 1. `lpc-support` 扩展

这是入口。

它表面上是一个 VS Code 扩展，但在这套范式里，它真正做的是三件事：

- 给项目建立工作区真源
- 给 LPC 提供结构化理解
- 把编译、错误树、包安装协议、AI 工作流统一到同一个上下文里

所以它的正确用法不是“先装扩展，再去设置页里填一堆路径”。

正确顺序应该是：

1. 在项目根目录建立 `lpc-support.json`
2. 让它指向真实使用的 `config.hell`
3. 让扩展同步 `resolved` 信息
4. 再决定当前项目走本地 `lpccp` 还是远程 HTTP

这时候扩展才不是孤立工具，而是整个工作区的入口。

### 2. 开发环境 driver 和 `lpccp`

只靠编辑器不够。开发环境这层还需要一个真正服务本地开发的 driver。

这也是为什么我维护了那个 FluffOS fork。

它不是一堆零散改动拼在一起，而是明确在补开发环境底层缺口：

- HTTP helper / parser efuns
- 本地 compile-service 工作流
- `lpccp`
- Windows `dist` 产物
- 编译器诊断可信度修复

现在获取这套 driver 有两条路。

第一条路，是直接使用我维护的 fork：

- [`https://github.com/lude2001/fluffos`](https://github.com/lude2001/fluffos)

如果你想自己编译开发驱动，而且不想和 FluffOS 官方主线互相干扰，最合适的方式就是基于这个仓库再 fork 一份，在你自己的 fork 里编译。

第二条路，是直接使用本仓库里提供的预编译 Windows 开发驱动：

- [`docs/Fluffos_Development_Drive/dist.zip`](D:/code/lpc-support/docs/Fluffos_Development_Drive/dist.zip)

解压以后，真正要关心的是两个东西：

- `driver`
- `lpccp`

这里有一点必须说清楚：`lpccp` 不是一个离线编译器。

它本质上是开发驱动的本地 compile-service 客户端。它要连接一个已经运行中的 driver，请求这个 driver 在当前运行环境里重新编译文件或目录，然后把结构化结果返回出来。

所以更稳妥的顺序应该是：

1. 先启动开发环境 driver
2. 再让 `lpccp` 连接它
3. 再让 `lpc-support` 通过 `compile.local` 使用它

如果这一点不理解清楚，后面的工作流就会一直拧巴。

### 3. `external_system_package`

这个东西也不是一个临时脚本目录。

按我现在的理解，它更像是：LPC 外部系统能力开始模块化的第一步。

它的重点不在于今天有哪个 HTTP 控制器，而在于它开始定义一套更明确的组织方式：

- package 有自己的目录
- package 有 `version.json`
- package 可以声明宏
- package 可以声明预加载对象
- package 可以有自己的安装协议
- AI 助手可以按协议自动安装

这件事的意义很大。因为一旦 package 开始变成标准能力，未来就不会只有一个 `external_system_package`。会有更多 package，而且它们不应该继续靠复制粘贴和口头说明传播。

它们应该靠：

- 元数据
- 安装协议
- 工作区真源
- AI 助手执行

来传播。

我理解这才是更值得继续往下推的方向。

## 这些资源怎么实际接起来

如果从最务实的角度讲，这套东西的落地顺序大概就是下面这样：

1. 准备一个 Mudlib 工作区
2. 在根目录建立 `lpc-support.json`
3. 让它指向真实使用的 `config.hell`
4. 安装 `lpc-support` 扩展
5. 准备开发环境 driver
6. 让 `lpccp` 可用
7. 在扩展里切换到本地编译模式
8. 如有需要，再安装 `external_system_package`
9. 开始在同一个工作区里做编辑、编译、诊断、错误查看和外部能力接入

这一步打通以后，你得到的就不再是几个分散工具，而是一条真正连起来的开发链路：

- 编辑器知道项目结构
- 编译知道项目配置
- 错误树知道当前编译环境
- `lpccp` 知道当前 driver
- package 知道自己该怎么安装进工程
- AI 助手知道它应该按什么协议工作

按我的理解，这一步比单独多支持几个语法点更重要。因为它改变的不是某一个局部体验，而是开发者和工程之间的关系。

过去很多事情靠人手工把链路串起来，现在开始变成系统自己知道该怎么协作。

## 为什么我说它想支撑更大的世界

如果按我的理解往下说，这套东西最终想支撑的，不是“更轻松地维护旧世界”，而是“更有把握地创造新世界”。

这里的重点不是题材新旧，而是组织方式。

一个真正大的 MUD 世界，不应该只是很多局部故事堆在一起。它需要：

- 更稳定的工程结构
- 更清晰的模块边界
- 更可靠的工具反馈
- 更低的协作摩擦
- 更自然的外部能力接入

如果这些基础没有，世界越大，代价越高。最后很多事情不是做不出来，而是维护方式越来越像手工拼接。

按我的判断，LPC 如果还想继续承担更大规模的世界创造，这层基础大概率还是要补上。

不是为了追求“现代化”这个词本身，而是因为没有这层基础，很多东西往后走会越来越重。

## 接下来还要继续做什么

现在这套东西当然还远远没有完成。

按我现在的判断，接下来最值得继续往下做的，还是三件事。

### 1. package 体系继续标准化

现在 `external_system_package` 只是第一个 package。

后面真正要做的是把 package 体系本身做实：

- 元数据字段怎么定义
- 宏注入怎么定义
- 预加载声明怎么定义
- AI 助手安装协议怎么定义
- 包升级和兼容怎么定义

只有这套东西稳定下来，LPC 工程能力才可能积累，而不是每次都从头手工接。

### 2. 开发驱动继续围绕本地平台建设

这个 fork 以后还会继续有改动，但我希望它越来越明确一件事：

它服务的不是“功能多一点”，而是“本地 LPC 应用开发更顺一点”。

也就是说，后续 driver 级改动都应该先问一句：

它有没有让 LPC 在开发阶段更像一个现代本地平台？

如果没有，那至少不是当前路线里的优先项。

### 3. 编辑器、driver、package、AI 协议继续打通

理想状态下，编辑器里的结构理解、driver 的编译反馈、package 的元数据、AI 的安装协议、工作区的工程真源，最后应该形成一条统一链路。

也就是说，项目不是在分别使用：

- 一个扩展
- 一个 driver
- 一些脚本
- 一些约定

而是在使用一套互相知道彼此存在的开发系统。

如果这一步做成，LPC 的开发体验就会真正开始脱离“靠经验维持的老环境”，转到“有基础设施支撑的新环境”。

## 相关项目推荐和分享

这里也顺手把几个我认为值得关注的项目放进来。

原因很简单：`lpc-support` 不是凭空冒出来的，也不是 LPC 现代化开发里唯一值得看的方向。相反，我一直认为，真正有价值的是大家分别从不同角度把这件事往前推。

### 1. `jlchmura/lpc-language-server`

项目地址：

- [`https://github.com/jlchmura/lpc-language-server`](https://github.com/jlchmura/lpc-language-server)

这是一个我很推荐去看的项目。

它体现出来的方向，是更强的 LPC 静态分析和语言服务能力。换句话说，它更像是在认真回答一个问题：如果把 LPC 当成一个现代语言来做语言服务器，这件事应该怎么往下做。

`lpc-support` 在 AST 相关工作上，确实参考过它的一些实现思路。但我要说明一点：两边走的具体理念并不完全一样。

我的路线更强调：

- 工作区工程真源
- 编辑器、driver、package、AI 工作流的统一
- 以开发环境闭环为核心来组织工具链

而 `lpc-language-server` 更鲜明的价值，在于它把 LPC 的静态分析能力往更强、更独立、更语言服务化的方向推了一步。

按我的看法，它值得被看到，也值得被持续参考。

### 2. `gesslar/lpc-mcp`

项目地址：

- [`https://github.com/gesslar/lpc-mcp`](https://github.com/gesslar/lpc-mcp)

这个项目和 `lpc-language-server` 是正相关的，可以把它理解成这条路线上的外部增强项。

如果说 `lpc-language-server` 更偏向语言服务本体，那 `lpc-mcp` 就是在往“让外部工具更容易接入 LPC 开发上下文”这件事上继续往前走。

这件事和我做 `external_system_package`、AI 安装协议、工作区真源，其实在大的方向上是相通的。大家切入点不一样，但都在做同一类基础工作：让 LPC 的开发能力不再封闭在老式手工流程里，而是能和现代工具链更自然地接起来。

### 3. `serenez/lpc-server-update`

项目地址：

- [`https://github.com/serenez/lpc-server-update`](https://github.com/serenez/lpc-server-update)

这个项目提供的是一套基于 socket 的 update 文件能力。

它和我现在做的本地 `lpccp` / compile-service 路线并不是同一实现，但它同样是在解决一个非常现实的问题：怎么让代码更新这件事不再完全停留在老的手工方式里。

如果放在具体使用场景里看，按我的理解，它会更适合 ZJMud 这一类 Mudlib 场景。

也就是说，这个项目的价值不在于它是不是“唯一正确方案”，而在于它已经把某一类真实工程场景里的更新问题，往前推成了一套可以落地的机制。

## 我怎么看这些项目和 `lpc-support` 的关系

我不想把 `lpc-support` 写成一个“别的都不重要，只有这条路线才对”的东西。

恰恰相反，我更愿意把它放在一个更实际的位置上看：

- `lpc-language-server` 往更强的静态分析和语言服务走
- `lpc-mcp` 往外部工具协同走
- `lpc-server-update` 往特定 Mudlib 场景的更新机制走
- `lpc-support` 则更强调工作区真源、开发闭环、driver 配套、package 协议，以及把这些东西接成一条统一开发链路

这些方向不是互斥的。

按我的理解，它们共同说明了一件事：LPC 不是不能现代化，问题更多还是有没有人愿意把这件事一层一层做下去。

## 最后

我不觉得 `lpc-support` 已经完成了什么大事。

它现在只是刚刚把方向立住。

但有一点我已经很确定：

如果 LPC 还想继续往前走，还想支撑更大的世界创造，可能就不能只靠兼容旧 lib 和经验维护来撑。开发环境这层基础，大概率还是要重建。

`lpc-support`、`lpc-language-server`、`lpc-server-update`、开发环境 FluffOS fork、`lpccp`、`external_system_package`、工作区级项目配置、结构化分析链路，这些东西放在一起，才构成了现在这批人在共同往前推的那套东西。

这不是在把旧工具重新做得更顺手一点。

它更像是在一点一点把 LPC 开发这件事，重新建立起新的底子。
