# Product Overview

## Product Purpose

LPC Support 是一个面向 VS Code 的 LPC / FluffOS 开发扩展，目标是让 LPC 项目在主流编辑器中获得接近现代语言工具链的开发体验。

它主要解决三类问题：

- LPC 缺少稳定、持续维护的现代编辑器支持，导致语法理解、跳转和补全能力不足
- FluffOS 项目往往依赖继承、宏、include、模拟函数库等约定，普通文本编辑器难以理解
- 开发者需要在本地编辑、远程编译、错误查看、函数文档查询之间频繁切换，效率低且容易出错

这个项目的核心价值不是“做一个通用 IDE 平台”，而是为 LPC / FluffOS 的真实开发流程提供足够深、足够实用的编辑器能力。

## Target Users

### Primary Users

1. **FluffOS / LPMud 项目开发者**
   - 需要稳定的语法支持、补全、定义跳转、引用查找和重命名能力
   - 需要理解继承链、include 文件、宏路径和 efun / simulated efun 文档

2. **MUD 项目维护者与服务器管理者**
   - 需要在编辑器内快速触发远程编译、查看编译错误和运行时错误
   - 需要在多服务器环境中切换配置，减少手工操作

3. **新加入 LPC 项目的成员**
   - 需要更低的学习门槛，包括语法高亮、函数文档、代码片段和即时诊断
   - 需要通过编辑器反馈理解项目约定，而不是依赖口头传承

### Secondary Users

1. **维护大型旧代码库的开发者**
   - 需要在不重写旧项目的前提下获得基本现代化工具支持

2. **希望逐步规范团队代码质量的负责人**
   - 需要通过静态检查、注释生成和统一工作流降低维护成本

## Key Features

1. **语言理解与编辑体验**
   - LPC 语法高亮、语义标记、代码折叠、文档大纲和代码片段
   - 基于 ANTLR4 和 AST 的补全、结构体成员补全、对象访问补全
   - 当前文件、继承文件、include 文件、宏定义与路径相关跳转

2. **代码诊断与导航**
   - 语法错误诊断
   - 未使用局部变量、未使用参数、未使用全局变量等静态检查
   - 宏使用、对象访问、局部变量声明位置等规则检查
   - 引用查找、当前文件范围重命名、变量列表查看

3. **FluffOS 工作流集成**
   - 编译当前文件
   - 批量编译目录中的 `.c` 文件
   - 编译服务器配置与切换
   - 错误诊断中心展示远程编译错误和运行时错误

4. **函数文档与知识辅助**
   - 内置标准 efun 文档
   - 本地 simulated efun 文档解析
   - 当前文件、继承文件、include 文件函数文档面板
   - Javadoc 风格注释生成与 GLM-4 辅助说明生成

5. **性能与维护辅助**
   - 解析缓存、性能统计与缓存清理
   - 文件夹扫描和继承关系扫描
   - 为大型 LPC 工程提供可持续的编辑体验，而不是只服务示例文件

## Business Objectives

- **把 VS Code 变成可用的 LPC 主力编辑器**：覆盖日常最常用的编辑、导航、诊断和编译链路
- **降低维护旧 MUD 工程的成本**：减少靠经验定位问题的时间，提升代码可读性和可维护性
- **降低新成员上手成本**：把文档、规则和常见操作前置到编辑器内
- **建立可持续迭代的语言工具基础**：通过语法、AST、缓存和模块化能力，为后续格式化、更多语义分析和更多方言支持打基础
- **服务实际社区需求而非空泛扩张**：优先解决 FluffOS 开发者在真实工作流中的阻塞点

## Success Metrics

- **核心工作流覆盖**：补全、跳转、悬停、诊断、编译这几条链路在典型 FluffOS 项目中可稳定使用
- **问题发现效率**：用户能在编辑器内定位大部分常见语法和静态问题，而不是依赖运行后排查
- **导航准确度**：定义跳转、include 解析、继承函数发现的误报和漏报持续下降
- **远程工作流可用性**：服务器配置、编译触发、错误树展示在真实项目中可重复使用
- **维护效率**：新增语言能力时能通过现有 parser / diagnostics / provider 结构快速落地并补充测试

## Product Principles

1. **FluffOS-first**
   - 优先围绕当前仓库实际支持的 FluffOS 工作流做深做稳
   - 对其他 LPC 方言保持扩展空间，但不以牺牲现有准确性为代价

2. **Editor-native workflow**
   - 关键反馈必须尽量留在 VS Code 内完成
   - 用户不应为了补全、文档、编译或错误定位频繁切换到外部工具

3. **Pragmatic intelligence**
   - 优先提供“足够准确且可解释”的能力，而不是华而不实的自动化
   - AST、宏解析、继承扫描等能力要直接服务真实代码场景

4. **Local-first and configurable**
   - 本地解析、本地文档、本地配置优先
   - 涉及网络和 AI 的能力应是可选增强，而不是基础功能前提

5. **Incremental modernization**
   - 尊重 LPC 旧项目的历史包袱
   - 用渐进方式引入诊断、注释、导航和规范化能力，避免强制用户重构现有代码

## Monitoring & Visibility

- **Primary surfaces**: VS Code 编辑器诊断、Hover、补全面板、函数文档面板、输出通道、错误树视图
- **Real-time updates**:
  - 文档变更后触发诊断刷新
  - 活动编辑器切换后刷新函数文档上下文
  - 切换错误服务器后刷新远程错误树
- **Key information surfaced**:
  - 当前文件静态诊断
  - 远程编译结果
  - 运行时错误列表
  - 解析缓存和性能统计
- **Sharing/export expectations**:
  - 当前项目暂无独立 Web 仪表板、报表导出或团队协作视图
  - 后续若新增此类能力，应作为明确的新产品能力，而不是默认假设

## Future Vision

### Near-term Direction

- 提升补全、跳转和诊断在复杂继承链、宏路径和 include 解析场景下的准确性
- 完善函数文档提取与注释生成体验
- 持续降低大型项目中的解析开销和误报率

### Potential Enhancements

- **Formatting**: 提供稳定的 LPC 代码格式化能力
- **Project intelligence**: 增强跨文件符号索引、继承关系可视化和更完整的引用分析
- **Dialect support**: 在不破坏 FluffOS 体验的前提下扩展到更多 LPC 方言
- **Server integration**: 增加更清晰的编译反馈、更多服务端接口兼容层和更好的错误聚合展示
- **Documentation workflows**: 改进函数注释生成、离线文档更新和项目内文档复用能力

### Long-term Vision

把 LPC Support 建成一个可靠、务实、可持续维护的 LPC 开发基础设施项目，让 LPC 团队能够在现代编辑器中完成主要开发工作，而不必为了语言限制退回到低效工具链。
